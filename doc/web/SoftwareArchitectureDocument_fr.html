Introduction

Objectif

Ce document fournit une vue d'ensemble de l'architecture complète du système, au travers d'une série de vues architecturales différentes dépeignant les différents aspects du système. Il vise à capturer et transmettre les décisions architecturales significatives qui sont prises pour le système.

Périmètre

Ce document s'applique au projet de Système d'Information RR0 (RR0 Information System ou UFO@home). Il affecte l'ensemble des besoins et de la conception de UFO@home.

Definitions, acronymes et abbréviations

  • IA— Intelligence artificielle.
  • API — Application Programming Interface : interface de programmation.
  • HTTP — Hyper-Text Transfer Protocol. Protocole de communication entre des navigateurs Web et des serveurs Web.
  • J2EE — Java 2 Enterprise Edition. Architecture Java de référence pour les applications serveur.
  • JDO — Java Data Objects. API Java standard pour gérer la persistance d'objets de manière transparente.
  • JTA — Java Transaction API : API Java pour les transactions.
  • RR0 IM— modèle d'information RR0 UFO@home.
  • Servlet — Java server process handling client (usually HTTP) requests.
  • UUDF — Unified Ufological Data Format : format de données ufologiques unifié.
  • UML — Unified Modeling Language.

See also the project's Glossary.

Objectifs et contraintes d'architecture

Certains besoins-clés et contraintes système ont un impact significatif sur l'architecture. Il s'agit du fait que :

  • un site Web UFO@home fournit un affichage de la plupart de son contenu. Une interface pour ce système doit être capable de supporter de grands volumes de trafic.
  • plusieurs serveurs UFO@home doivent être capables de communiquer via Internet pour échanger des données ou permettre un traitement distribué.
  • toute personne ou organisation doit pouvoir implémenter sa propre version d'un serveur compatible UFO@home, lui permettant ainsi de communiquer (échanger des données, etc.) avec d'autres serveurs compatibles UFO@home au travers d'Internet.
  • toutes les fonctions doivent être accessibles à l'aide des trois navigateurs Web les plus utilisés.
  • touts les impératifs de performance et de charge, tels qu'ils sont stipulés dans le document de Vision [1] et dans les Spécifications supplémentaires [2], doivent être pris en consideration lors du développement de l'architecture.

Vue d'ensemble

Cette sous-section décrit la décomposition de l'ensemble du modèle de conception en termes de hiérarchie et packages et de couches.

En résumé, UFO@home peut être vu comme l'empilage de 3 sous-systèmes :

  1. le Système d'Information (IS), qui est l'application accessible aux utilisateurs finaux du public. Elle gère les interactions avec les utilisateurs, leurs contextes de travail, et fournit des services spécifiques à cette application tels que le calcul de statistiques, etc.
  2. le Modèle d'Information (IM), qui modélise les concepts du domaine ufologique, qu'ils soient de haut (cas, témoignanges, classifications) ou de bas (événements, moments) niveau.
  3. le moteur Prosper (peut-être), qui ne fait pas partie du projet UFO@home, mais est un projet indépendant visant à offrir des capacités d'intelligence artificielle (AI) à toute application, permettant de prédire des faits futurs sur la base de faits passés, d'identifier des motifs d'événements, etc. UFO@home étant un bon candidat pour expérimenter Prosper, ce dernier a été identifié comme un composant pluggable et intéressant pour UFO@home.

UFO@home est construit sur une architecture 3 couches séparant les préoccupation de présentation, métier et d'intégration/accès aux données. Ceci permet de supporte plusieurs types de clients, tels que des navigateurs Web, des clients graphiques "lourds", et même d'autres systèmes d'informations à l'aide de services Web.

classification

Le sous-package de classification définit un ensemble réduit de classes permettant de définir n'importe quelle classification de donneés. Il permet de classer des Cas, Observations (RR1, RR2...), mais aussi d'autres objets métier tels que des Nuages, Températures (Froid, Chaud...), etc. Un avantage important de ce système de classification est de ne pas se baser sur une classification prédéfinie, permettant ainsi à tout utilisateur de bâtir son propre système de classification de tout type de données.

Vue du déploiement

Cette section décrit une ou plusieurs configurations réseau (matérielles) sur laquelle le logiciel est déployé et s'exécute. Pour chaque configuration elle indique quels noeuds physiques (ordinateurs, processeurs) exécutent le logiciel, ainsi que leurs interconnections (bus, réseau local, point-à-point, etc.). Elle inclut également une correspondance de la Vue des Processus sur les noeuds physiques.

Un serveur UFO@home peut s'exécuter sur toute plate-forme compatible Java (i.e. Windows, Unix, et d'autres). Une machine cliente est tout appareil capable d'exécuter un navigateur Web (très probablement un PC, mais pas nécessairement) et de se connecter à un serveur UFO@home via Internet.

Plusieurs serveurs UFO@home peuvent communiquer via Internet en utilisant des services Web. De telles communications peuvent intervenir lors d'échanges de données (exploitant UUDF), traitements/calculs distribués, etc..
